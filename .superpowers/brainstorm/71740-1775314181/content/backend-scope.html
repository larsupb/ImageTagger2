Backend Multi-Project Design

The backend currently supports multiple concurrent sessions in memory, but has no API to manage or enumerate them. What level of multi-project support do you want?

POST /api/projects/open
Creates session + loads dataset in one call
GET /api/projects
List active sessions with metadata
DELETE /api/projects/:id
Close a session

A: Project Management API

Add 3 new endpoints: open (create + load in one call), list, close. Existing endpoints unchanged. Sessions remain in-memory. Background cleanup task for expired sessions. Minimal changes to existing code.

Session metadata
Each session stores project name, path, masks path, created time
Background cleanup
Periodic task removes expired sessions
Session persistence (optional)
Save session list to disk, restore on restart

B: Full Project Model

Enrich sessions with project metadata (name, display name, last accessed). Add background cleanup task. Optionally persist session list to disk for recovery on restart. New endpoints like A but with richer project info.

Project config files
.imagetagger project files in dataset directories
Recent projects list
Track recently opened projects globally
Project-specific settings
Each project can override global settings

C: Project Files + Persistence

Introduce project config files (.imagetagger) in dataset directories. Track recent projects. Support project-specific settings overrides. Full session persistence. Most complete but most complex.
